This guide covers the full Acton workflow: scaffolding a counter contract, compiling it, linting and formatting code, running tests with coverage and mutation analysis, setting up a wallet, deploying to testnet, and verifying the source.

<Callout type="info" title="Already have a repository?">
    This quickstart uses `acton new` because it is the fastest way to get a working sample project. If adding Acton to an existing repository instead, start with `acton init` and then return to the building, testing, and deployment steps below. The existing project flow is documented in [project management](/docs/projects).
</Callout>

## Prerequisites

- [Acton](/docs/installation) 0.4.0 or later installed.
- [Node.js](https://nodejs.org/en/download/) 22 or later LTS: required for the TypeScript app scaffold added by `--app`.

## Initialize a new project

`acton new` scaffolds a new project directory from a built-in template. The `counter` template ships with a ready-made contract, tests, and a deployment script. The `--app` flag adds a Vite-based React frontend scaffold and generated TypeScript wrappers alongside the Tolk sources.

```bash
acton new first_counter --template counter --app
```

Output:

```acton-cli
$ acton new first_counter --template counter --app
  Installing standard library 0.4.0
> Initialized empty Git repository in /home/you/first_counter/.git/
✓ Created new Acton project
  Project name: first_counter
  Description: A TON blockchain project
  Template: counter
  TypeScript app: included
  License: MIT

Created Acton.toml with project configuration
```

Navigate to the project folder:

```bash
cd first_counter
```

With `--app`, the project layout places Tolk sources under `contracts/src/`, tests under `contracts/tests/`, and scripts under `contracts/scripts/`. Generated TypeScript wrappers go into `wrappers-ts/`, and the React frontend lives in `app/`. Install the npm workspace dependencies before running any frontend commands:

```bash
npm ci
```


The generated `Acton.toml` includes ready-to-run script aliases `deploy-emulation` and `deploy-testnet`. The emulation alias runs the deployment script entirely against a local in-process node — no wallet, no network — and is the fastest way to confirm everything wires up correctly:

```bash
acton run deploy-emulation
```

Output:

```acton-cli-trace
$ acton run deploy-emulation
   Compiling contracts
    Finished in 712.5ms
N/A -> deployer A
└── empty 0.005 TON -> Counter B                                                gas=692
    └── account created

Deployed counter to EQB...m6n (Counter)
Counter owner is EQB...m6n (deployer)
Counter value is 0
```

Use `acton run deploy-testnet` once a funded wallet is set up — see [Deploy to testnet](#deploy-to-testnet) below.

Other useful `acton new` flags:

| Flag | Purpose |
|------|---------|
| `--template <name>` | Select a template: `empty`, `counter`, `jetton`, `nft`, or `w5-extension` |
| `--app` | Include the Vite + React frontend scaffold and TypeScript wrappers |
| `--hooks` | Install Git pre-commit hooks for format and lint checks |
| `--agents` | Add an `AGENTS.md` file with coding-agent guidance |
| `--license <id>` | Set the project license: `MIT`, `Apache-2.0`, `GPL-3.0`, `BSD-3-Clause`, `ISC`, `Unlicense`, or `Other` |

## Build the contract

`acton build` compiles every contract configured in `Acton.toml` and places build artifacts in `build/`.

```bash
acton build
```

Output:

```acton-cli
$ acton build
   Compiling contracts
   Compiling Counter
    Finished in 4.8ms
    Finished in 728.8µs
```

Each compiled contract produces a JSON artifact:

```json title="build/Counter.json"
{
  "code_boc64": "te6ccgEBBAEAbwABFP8A9KQT9Lzy...",
  "hash": "1F6548A25E04497496242AD7804B653E6DA9A47E474DC17436...."
}
```

Build a single contract by name:

```bash
acton build Counter
```

Other useful build flags:

| Flag | Purpose |
|------|---------|
| `--info` | Print resolved paths, configured contracts, and environment variables without compiling |
| `--clear-cache` | Discard the build cache and recompile from scratch |
| `--graph deps.dot` | Write a DOT dependency graph to `deps.dot` for visualization |
| `--output-abi <dir>` | Also write ABI JSON files to the given directory |
| `--output-fift <dir>` | Also write compiled Fift files to the given directory |

Print the resolved project configuration without triggering a build:

```bash
acton build --info
```

Output:

```acton-cli
$ acton build --info
   Compiling contracts
   Compiling Counter
    Finished in 4.6ms
    Finished in 730.0µs

   Artifacts of Counter
        Code te6ccgEBBAEAbwABFP8A9KQT9LzyyAsBAgEgAgMAAtIA...
        Hash 0x1F6548A25E04497496242AD7804B653E6DA9A47E474DC174364...
```

## Format and check code

### Format Tolk files

`acton fmt` formats all `.tolk` files under the project root using the built-in Tolk formatter.

```bash
acton fmt
```

Output:

```acton-cli
$ acton fmt
All files are properly formatted
```

Run in check mode to detect formatting violations without rewriting files (useful in CI):

```bash
acton fmt --check
```

When a file is not formatted correctly, `--check` prints a unified diff and exits with a non-zero status:

```acton-cli
$ acton fmt --check
Diff in contracts/src/Counter.tolk:
-fun onInternalMessage(in:InMessage){
-val x=1;
+fun onInternalMessage(in: InMessage) {
+    val x = 1;
     val y = 2;
 }
```

Format a specific file or directory:

```bash
acton fmt contracts/src/Counter.tolk
```

Format also works in IDE with default hotkeys.

### Lint Tolk contracts

`acton check` runs the Tolk linter across the whole project and reports diagnostics. On a clean project, it prints nothing and exits with status 0. When issues are found, output looks like this:

```acton-cli-check
$ acton check
warning[unused-variable]: variable `x` is unused
  --> contracts/src/Counter.tolk:12:9
   |
12 |     val x = 1;
   |         ^ help: prefix with `_` to silence

1 issue can be fixed automatically, rerun with --fix flag.

Use acton check --explain <CODE> to get detailed explanation of a rule.
For example: acton check --explain unused-variable
```

Check a specific contract by name or file path:

```bash
acton check Counter
```

## Regenerate wrappers

`acton wrapper` generates typed Tolk wrappers from the contract's compiler ABI. A prior `acton build` run is not required.

Regenerate wrappers for every contract at once:

```bash
acton wrapper --all
```

Output:

```acton-cli-wrapper
$ acton wrapper --all
   Generated wrappers/Counter.tolk
   Generated contracts/tests/Counter.test.tolk
```

Generate the wrapper for a single contract:

```bash
acton wrapper Counter
```

Pass `--ts` to emit a TypeScript wrapper instead, which the app scaffold consumes from `wrappers-ts/`:

```bash
acton wrapper Counter --ts
```

```acton-cli-wrapper
$ acton wrapper Counter --ts
   Generated wrappers-ts/Counter.ts
```

## Test the contract

The `acton test` discovers and runs all files matching `*.test.tolk`.

The template includes `contracts/tests/counter.test.tolk`, which checks the counter increase rate.

To run all tests:

```bash
acton test
```

Output:

```acton-cli
$ acton test
   Compiling contracts
    Finished in 726.375µs
     Running tests

 TEST  <root>/first_counter

 > ./contracts/tests/counter.test.tolk (3 tests)
  ✓ increase-counter 4ms
  ✓ reset-counter 3ms
  ✓ unknown-message 2ms

 ✓ 3 passed in 1 file
```

Other useful test flags:

| Flag | Purpose |
|------|---------|
| `--filter <pattern>` | Run only tests whose name matches the pattern |
| `--fail-fast` | Stop after the first failure |
| `--save-test-trace` | Write TVM execution traces to `build/traces/` |
| `--fork-net testnet` | Resolve remote contract and account state from testnet during tests |

### Collect coverage

`--coverage` collects line and branch coverage and prints a summary after the run:

```bash
acton test --coverage
```

Set a minimum threshold to fail CI when coverage drops below a target:

```bash
acton test --coverage --coverage-minimum-percent 80
```

### Inspect results in the browser UI

`--ui` opens an interactive browser interface while tests run. Combine it with `--coverage` to browse annotated source files in the Coverage tab:

```bash
acton test --ui --coverage
```

### Run mutation testing

Coverage shows which lines ran — not whether the tests would catch a bug on those lines. Mutation testing fills the gap: it makes small, deliberate changes to the contract — for example, replacing `+= 1` with `-= 1` or removing an `assert` — and reruns the suite. If every test still passes against a mutation, that mutation is a **survivor** — the line is exercised but its behavior is not actually verified.

Use `--mutate-contract` to restrict mutations to a single contract:

```bash
acton test --mutate --mutate-contract Counter
```

Output:

```acton-cli-mutate
$ acton test --mutate --mutate-contract Counter

Mutation Testing
────────────────────────────────────────────────────────────
Session:  20260507-103452-7d8c
Contract: Counter
Source:   contracts/src/Counter.tolk
Files:    1
Mutants:  6

  ◉ Mutation 1/6 contracts/src/Counter.tolk:18 Replace == with != KILLED
  ◉ Mutation 2/6 contracts/src/Counter.tolk:22 Replace `+=` with `-=` KILLED
  ◉ Mutation 3/6 contracts/src/Counter.tolk:22 Remove save() method calls SURVIVED
  ◉ Mutation 4/6 contracts/src/Counter.tolk:28 Replace `true` with `false` KILLED
  ◉ Mutation 5/6 contracts/src/Counter.tolk:30 Remove assert statements KILLED
  ◉ Mutation 6/6 contracts/src/Counter.tolk:33 Remove throw keyword KILLED

    Total mutants        6
  ✓ Killed               5
  ✗ Survived             1
  ! Compile errors       0

  ◆ Mutation Score       83.3%
```

Each survivor pinpoints a specific line and rule. Add a test that exercises the corresponding path and rerun — the mutation will be killed on the next run.

<Callout type="tip">
  Use `--mutation-diff branch` to test only mutations on lines changed since the merge base — useful in CI to skip re-testing stable code. See the [mutation testing docs](/docs/testing/mutation-testing/overview) for details.
</Callout>

## Debug the contract

Acton ships a source-level debugger that uses the [Debug Adapter Protocol](https://microsoft.github.io/debug-adapter-protocol/) (DAP). It runs the same contract code as production — no separate debug build — and exposes source breakpoints, stepping, stack traces, locals, and the live `c4` / `c5` / `c7` registers.

The `--debug` flag is available on three commands:

| Command | What gets debugged |
|---------|--------------------|
| `acton test [path] --debug` | A live local test session — combine with `--filter` to narrow down to one test |
| `acton script <path> --debug` | A locally executing script |
| `acton retrace <HASH> --contract <NAME> --debug` | Debug a real on-chain transaction |

### Start a debug session from your editor

Acton works best with the [TON VS Code extension](/docs/ide-support/vscode) and the [TON JetBrains plugin](/docs/ide-support/jetbrains), which ship a Tolk debug adapter type out of the box. Other DAP clients can also attach.

#### JetBrains IDEs

Open a `*.test.tolk` file and use the run-configuration **Debug** button. The plugin starts Acton with `--debug` and attaches automatically — no launch configuration to edit.

#### VS Code

Two options, depending on how the test run starts:

1. **Debug Test from the editor.** Right-click a test inside a `*.test.tolk` file and pick **Debug Test**. The TON extension launches Acton with `--debug` and attaches in one step.
2. **Attach to a running Acton session.** First start Acton with `--debug` and a fixed port:

```bash
acton test contracts/tests/counter.test.tolk --filter "increase counter" --debug --debug-port 4711
```

Output:

```acton-cli
$ acton test contracts/tests/counter.test.tolk --filter "increase counter" --debug --debug-port 4711
  Compiling contracts
   Finished in 6.2ms
    Running tests
 Debug adapter listening on 127.0.0.1:4711
 Waiting for client to connect...
```

Acton blocks until a client attaches. Add the following to `.vscode/launch.json`:

```json title=".vscode/launch.json"
{
 "version": "0.2.0",
 "configurations": [
   {
     "type": "tolk",
     "request": "launch",
     "name": "Retrace DAP",
     "debugServer": 4711
   }
 ]
}
```

Once the terminal shows `Waiting for client to connect...`, open the **Run and Debug** panel, pick **Retrace DAP** from the configuration dropdown and run it. VS Code connects to the listening port and the test launches under the debugger.

The same configuration works for `acton script --debug` and `acton retrace --debug` — just match the `--debug-port` value to `debugServer`.

With the editor attached, set breakpoints in `.tolk` sources, and step through code as the test runs.

### What the debugger shows

- **Variables** — see every local at the current frame and watch how its value changes between steps.
- **Lazy objects** — `lazy` values are inspectable even when their fields have not been loaded from storage yet.
- **Stepping** — step over, step into, and step out, including across inlined function boundaries.
- **Breakpoints** — set source breakpoints or use *run to cursor* to advance to a chosen line.
- **Exception breaks** — stop automatically when an uncaught exception is thrown.

Everything behaves exactly as expected from a source-level debugger.

Also, you can get source locations for a failure without stepping via `--backtrace full`:

```bash
acton test --backtrace full
```

Output (for a failing test):

```acton-cli-trace
$ acton test --backtrace full

    Finished in 5.1ms
    Finished in 731.2µs

        └── IncreaseCounter 0.1 TON -> Counter B           gas=1513 exit_code=10 aborted
```

## Set up a wallet

Deployment requires a funded wallet. Create one and request testnet funds in a single step:

```bash
acton wallet new --name deployer --local --airdrop
```

Output:

```acton-cli
$ acton wallet new --name deployer --local --airdrop

✓ Wallet successfully created and added to wallets.toml
✓ Wallet address is EQB...m6n

✓ The mnemonic is securely stored in your system's keyring.
→ Requesting airdrop for wallet deployer EQB...m6n

To check wallet balances run acton wallet list --balance.
```

This creates a new wallet and saves its configuration to `wallets.toml` in the project root. By default, Acton stores the mnemonic in the system keyring when secure storage is available.

<Callout type="warn">
    If secure storage is unavailable, or if disabled with `--secure false`, the mnemonic is written to `wallets.toml`. Never commit it to git. Acton adds it to `.gitignore` automatically.
</Callout>

The `--airdrop` flag automatically requests testnet funds from the configured [faucet](/docs/wallets#fund-a-wallet-on-testnet) after the wallet is created.

The full wallet subcommand reference:

| Command | Purpose |
|---------|---------|
| `acton wallet new --name deployer --local --airdrop` | Create a wallet and request testnet funds |
| `acton wallet import --name deployer` | Import an existing mnemonic |
| `acton wallet list --balance` | List configured wallets and their on-chain balances |
| `acton wallet airdrop deployer --net testnet` | Request testnet funds for an existing wallet |
| `acton wallet sign deployer --body <hex>` | Sign an external message body without running a script |
| `acton wallet export-mnemonic deployer` | Export the mnemonic interactively |
| `acton wallet remove deployer -y` | Remove a wallet non-interactively |

The full reference is in [Setup wallets](/docs/tutorial/setup-wallets) and [`acton wallet`](/docs/commands/wallet).

## Deploy to testnet

The template includes the deployment script: `contracts/scripts/deploy.tolk`. To deploy in testnet:

```bash
acton script contracts/scripts/deploy.tolk --net testnet
```

If successful, the output includes the deployment transaction hash, the contract address, and the freshly read on-chain counter value.

The `Acton.toml` script alias `deploy-testnet` wraps this command, so the following is equivalent:

```bash
acton run deploy-testnet
```

### Sign with a browser wallet via TON Connect

Pass `--tonconnect` to sign transactions through a connected TON Connect wallet:

```bash
acton script contracts/scripts/deploy.tolk --net testnet --tonconnect
```

Acton saves the TON Connect session to `build/sessions/tonconnect/<network>.json` and reuses it on subsequent runs.

### Fork network state locally

`--fork-net` resolves live account and contract state from a network while keeping execution local — no transactions are broadcast:

```bash
acton script contracts/scripts/query.tolk --fork-net testnet
```

### Debug script execution

Pass `--verbose` to print executor logs:

```bash
acton script contracts/scripts/deploy.tolk --net testnet --verbose
```

Pass `--show-bodies` to print decoded message bodies alongside each transaction in the output.

## Inspect remote state

`acton rpc` queries blockchain account state through a configured network endpoint. It works without a project manifest for raw inspection.

Check whether the deployed counter is active, read its balance, and match the on-chain code against the local build:

```bash
acton rpc info <CONTRACT_ADDRESS> --net testnet
```

Replace `<CONTRACT_ADDRESS>` with the address printed during deployment. When the on-chain `code_hash` matches a compiled contract in the local project, `acton rpc info` also prints the matched contract name and decodes account storage through the ABI.

Fetch the latest masterchain block sequence number:

```bash
acton rpc block-number --net testnet
```

Render a transaction trace as a decoded tree:

```bash
acton rpc trace <TX_HASH> --net testnet
```

## Verify the contract

`acton verify` checks that the deployed bytecode matches the local sources. It compiles local sources, uploads them to the verifier backend, collects the required signatures, and submits the verification transaction.

```bash
acton verify Counter --net testnet --address <CONTRACT_ADDRESS>
```

Replace `<CONTRACT_ADDRESS>` with the address from the deployment step.

Use `--dry-run` to compile and upload sources without submitting the final verification transaction:

```bash
acton verify Counter --net testnet --address <CONTRACT_ADDRESS> --dry-run
```

<Callout type="info">
    Verification requires a funded wallet. `--dry-run` skips the on-chain verification transaction and does not consume funds.
</Callout>

## Look up TVM instructions

`acton doc tvm` looks up TVM instruction documentation directly from the terminal:

```bash
acton doc tvm PUSHINT
```

Use `--find` for fuzzy search by keyword:

```bash
acton doc tvm --find "stack push"
```

## Diagnose the environment

`acton doctor` prints a diagnostic report covering resolved paths, manifest status, stdlib health, overlay status, native library metadata, and relevant environment variables. Run it before filing a bug report.

```bash
acton doctor
```
