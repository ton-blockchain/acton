Acton projects are defined by `Acton.toml`. That manifest tells Acton where contracts live, how they depend on each other, where generated files go, and which path aliases the compiler should resolve. For the full field-by-field reference, see [`Acton.toml`](/docs/acton-toml).

## Start from a new or existing directory

Use `acton new` when the project does not exist yet and a scaffold is useful.

```bash
acton new <PATH>
```

Use `acton init` when the repository already exists and only needs an Acton manifest, standard library files, and default project mappings.

```bash
acton init
```

`acton init` works in place. When `Acton.toml` does not exist, it scans the current directory tree for `.tolk` files that define `onInternalMessage` and registers them as contracts. When `Acton.toml` already exists, it does not recreate the project. It only patches missing default import mappings, refreshes `.acton/tolk-stdlib`, and updates `.gitignore` with the usual Acton entries.

`acton init` is safe to rerun after moving files around or after removing default mappings by accident.

<Callout type="caution">
  On an existing project, `acton init` reloads and writes `Acton.toml` again.
  Supported settings are preserved, but TOML comments and unknown keys are not.
</Callout>

## Update project metadata

Edit `[package]` for the metadata that identifies the project outside the contract graph.

```toml
[package]
name = "my-acton-project"
description = "A TON blockchain project"
version = "0.1.0"
repository = "https://github.com/example/my-acton-project"
license = "MIT"
```

This section is the right place for version bumps, repository changes, and license metadata. It is not used to register contracts.

## Add a new contract

To add a contract to an existing project, do two things:

- add the `.tolk` source file
- register it under `[contracts]` in `Acton.toml`

Example:

```toml
[contracts.counter]
display-name = "Counter"
src = "contracts/Counter.tolk"
depends = []
```

The contract key, `counter` in this example, is the contract ID. Acton commands use that ID, not the display name.

```bash
acton build counter
acton wrapper counter
```

Use `display-name` only for logs and UI. Keep the contract ID stable once other files or scripts start referring to it.

<Callout type="important">
  `display-name` is not a command name. `acton build Counter` fails if the
  manifest key is `counter`.
</Callout>

## Choose a good contract ID

Treat the manifest key as the durable identifier for the contract. Prefer a short, lowercase, machine-friendly name:

```toml
[contracts.jetton_minter]
display-name = "Jetton Minter"
src = "contracts/JettonMinter.tolk"
depends = ["jetton_wallet"]
```

This keeps command lines predictable and makes dependency declarations easier to read. The display name can stay human-readable and change later without breaking command usage.

## Move or rename a contract file

When a contract source file moves, update `src` in `Acton.toml`.

```toml
[contracts.counter]
display-name = "Counter"
src = "contracts/src/counter.tolk"
depends = []
```

When only the file path changes, keep the contract ID if possible. That avoids unnecessary edits in scripts, tests, CI, and other contracts that reference the ID in `depends`.

When the contract itself is being renamed, update:

- the source file path in `src`
- the contract header inside the `.tolk` file, if needed
- any `depends` entries that refer to the old contract ID
- wrapper imports or scripts that use the old generated file name

If the project layout changed substantially, rerun `acton init` to restore missing default mappings and refresh project glue.

## Add another contract to the same project

Multi-contract projects are a normal Acton layout. Add each contract as a separate entry under `[contracts]`.

```toml
[contracts.jetton_minter]
display-name = "Jetton Minter"
src = "contracts/JettonMinter.tolk"
depends = ["jetton_wallet"]

[contracts.jetton_wallet]
display-name = "Jetton Wallet"
src = "contracts/JettonWallet.tolk"
depends = []
```

By default, `acton build` compiles every configured contract. To work on one contract at a time, pass its contract ID:

```bash
acton build jetton_minter
```

Acton then builds that contract and its transitive dependencies.

## Declare contract dependencies

Use `depends` when one contract needs another contract's compiled code or a library reference during the build.

The short form is enough in most projects:

```toml
[contracts.root]
src = "contracts/Root.tolk"
depends = ["wallet"]
```

Use the detailed form when the generated helper needs a custom function name, dependency kind, or output path:

```toml
[contracts.root]
src = "contracts/Root.tolk"
depends = [
  { name = "wallet", kind = "library_ref", function = "get_wallet_code" }
]
```

Dependency names always refer to contract IDs from `Acton.toml`, not display names.

## Control where build artifacts go

Use `[build]` when the default `build/` and `gen/` directories no longer fit the project layout.

```toml
[build]
out-dir = "artifacts"
gen-dir = "artifacts/gen"
output-fift = "artifacts/fift"
```

This is useful when:

- CI expects artifacts in a fixed directory
- generated dependency helpers should stay out of the source tree
- compiled Fift output should be preserved explicitly

Per-contract `output` is separate from `[build]`. Use it when a compiled `.boc` must land at an exact path:

```toml
[contracts.wallet]
src = "contracts/Wallet.tolk"
output = "artifacts/wallet.boc"
depends = []
```

## Keep imports stable as the project grows

Use `[import-mappings]` to avoid brittle relative imports and to preserve a clean layout as directories move.

```toml
[import-mappings]
acton = ".acton"
contracts = "contracts/src"
tests = "contracts/tests"
wrappers = "contracts/wrappers"
gen = "gen"
```

This allows imports such as:

```tolk
import "@contracts/types"
import "@wrappers/Counter"
import "@acton/testing/expect"
```

Projects created with `acton new` and patched by `acton init` usually include the default mappings already. Edit them when the project layout changes. Keep `@acton` in place unless the standard library location is being changed deliberately.

## Change wrapper output paths

Use `[wrappers]` when generated wrappers should live in a stable location.

```toml
[wrappers.tolk]
output-dir = "wrappers"
generate-test = true
test-output-dir = "tests"

[wrappers.typescript]
output-dir = "wrappers-ts"
```

This is useful when wrappers are committed to the repository or imported by app code. It also avoids repeating `--output-dir` on every `acton wrapper` run.

For Tolk wrappers, Acton falls back to the `@wrappers` mapping when `[wrappers.tolk].output-dir` is not configured. Keep those two settings aligned if both are present.

## Add project scripts

Use `[scripts]` for repeated project commands.

```toml
[scripts]
deploy = "acton script scripts/deploy.tolk"
check = "acton test --filter=.*_check"
```

This keeps routine tasks in the manifest instead of spreading them across shell history or ad hoc README snippets.

## Know what not to edit manually

Do not treat everything in the project root as source of truth.

- `Acton.toml` is the manifest to edit directly.
- `.acton/` is Acton-managed internal state and standard library content.
- `build/` and `gen/` are generated outputs unless the manifest redirects them.

Keep generated directories in `.gitignore` unless the repository has a clear reason to commit them.

## Use `acton init` as a repair tool

Rerun `acton init` when the project needs a light reset without re-scaffolding the repository.

Common cases:

- `Acton.toml` is missing the default import mappings
- `.acton/tolk-stdlib` needs to be refreshed
- `.gitignore` no longer contains the standard Acton entries
- contract files were added before the first manifest was created

`acton init` does not replace an existing project structure. It fills in the missing Acton-specific pieces.
