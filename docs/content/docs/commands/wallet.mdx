---
title: "Managing Wallets"
description: "Learn how to create and manage testnet wallets for your Acton projects"
---

The `wallet` command allows you to generate, import, list, sign messages with, export mnemonics from, and remove wallets for testing purposes. It automatically creates wallet configurations and offers to store mnemonics securely in the system keyring or in configuration files.

<Callout type="warn">
    Wallets created by this command are configured for **testnet** by default.
    Coins in testnet have **NO VALUE**.
</Callout>

## Commands

### `acton wallet new`

Generates a new testnet wallet.

```bash
acton wallet new
```

This command will start an interactive prompt to guide you through the wallet creation process.

#### Options

| Option                | Type   | Description                                                                     |
|-----------------------|--------|---------------------------------------------------------------------------------|
| `--name <NAME>`       | String | Name of the wallet (default: prompts user)                                      |
| `--version <VERSION>` | String | Version of the wallet (default: prompts user)                                   |
| `--local`             | Flag   | Save wallet to local `wallets.toml` (mutually exclusive with `--global`)        |
| `--global`            | Flag   | Save wallet to global `global.wallets.toml` (mutually exclusive with `--local`) |
| `--secure`            | Bool   | Use secure native store (Keyring) (default: true if available)                  |
| `--airdrop`           | Flag   | Request testnet TON from faucet after wallet creation (default: false)          |
| `--json`              | Flag   | Output result as JSON                                                           |

<Callout type="info">
Path behavior:

- Local `wallets.toml` and local `global.wallets.toml` symlink are created in project root for the current invocation.
- Global project root/manifest resolution is described in
  [`Project Root Resolution`](/docs/acton-toml/#project-root-resolution).
- Use `--project-root <PATH>` when running from another directory and you want wallet files to be written into that project.
</Callout>

**Supported wallet versions:**
- `v5r1` (Default)
- `v4r2`
- `v3r2`
- `highloadv2r2`
- And other historical versions (`v1r1`-`v3r1`, `highloadv1r1`-`highloadv2r1`)

#### Examples

**Interactive Creation:**

Simply run the command without arguments to use the interactive mode:

```bash
acton wallet new
```

You will be prompted to:
1. Enter a name for your wallet
2. Choose where to save it (local or global)
3. Select the wallet version from a list
4. Choose whether to store the mnemonic in the system's secure keyring (recommended)
5. Choose whether to request testnet TON from faucet automatically

**Non-interactive Creation:**

```bash
# Create a v5r1 wallet named "deployer" locally
acton wallet new --name deployer --version v5r1 --local

# Create a global highload wallet
acton wallet new --name shared-sender --version highloadv2r2 --global

# Create wallet and request testnet TON from faucet immediately
acton wallet new --name deployer --version v5r1 --local --airdrop
```

**Wallet name normalization:**

Wallet names are normalized automatically:
- converted to lowercase
- spaces are converted to `-`
- only letters, digits, `-`, and `_` are kept

Example: `My Wallet 123!` becomes `my-wallet-123`.

**`--secure` behavior:**

- If secure native store is available and `--secure` is omitted, Acton prompts and defaults to secure storage.
- If secure native store is unavailable and `--secure=true` is passed explicitly, the command fails.
- If secure native store is unavailable and `--secure` is omitted (or `false`), mnemonic is stored in plain text config.

**`--airdrop` behavior:**

- Default is `false`.
- In interactive mode, if `--airdrop` is not passed, Acton asks whether to request testnet TON automatically.
- In non-interactive mode, `--airdrop` must be passed explicitly to trigger auto-request.

#### Output

When you create a new wallet, the configuration is added to either `wallets.toml` (local) or `~/.config/acton/wallets/global.wallets.toml` (global).

**Example (Plain Text):**
```toml
[wallets.deployer]
kind = "v5r1"
workchain = 0
keys = { mnemonic = "word1 word2 ... word24" }

[wallets.deployer.expected]
address-testnet = "EQ..."
```

**Example (Secure Keyring):**
```toml
[wallets.deployer]
kind = "v5r1"
workchain = 0
keys = { mnemonic-keyring = "my-project:deployer" }

[wallets.deployer.expected]
address-testnet = "EQ..."
```

For local wallets, keyring id usually includes project prefix (`<project>:<wallet>`).
For global wallets, keyring id is wallet name.
`wallet new` automatically writes only `expected.address-testnet`; add `address-mainnet` manually if needed.
When a wallet is saved globally, Acton may also create a local `global.wallets.toml` symlink in project root for the current invocation.

In `--json` mode, output includes:
- `airdrop_requested` (boolean)
- optional `airdrop` object when auto-request was attempted (`--airdrop` or interactive confirmation)

### `acton wallet import`

Imports an existing wallet from a mnemonic phrase.

```bash
acton wallet import <MNEMONICS>...
```

This command allows you to import an existing wallet by providing its mnemonic phrase.

#### Options

| Option                | Type   | Description                                                    |
|-----------------------|--------|----------------------------------------------------------------|
| `--name <NAME>`       | String | Name of the wallet (default: prompts user)                     |
| `--version <VERSION>` | String | Version of the wallet (default: prompts user)                  |
| `--local`             | Flag   | Save wallet to local `wallets.toml` (mutually exclusive with `--global`) |
| `--global`            | Flag   | Save wallet to global `global.wallets.toml` (mutually exclusive with `--local`) |
| `--secure`            | Bool   | Use secure native store (Keyring) (default: true if available) |
| `--json`              | Flag   | Output result as JSON                                          |

#### Examples

**Interactive Import:**

```bash
acton wallet import
```

You will be prompted to enter your mnemonic phrase, choose a name, and select where to save it.

**Non-interactive Import:**

```bash
# Import a wallet named "my-wallet" locally
acton wallet import --name my-wallet --local "word1 word2 ... word24"

# Import a wallet with specific version
acton wallet import --name deployer --version v4r2 --local "word1 word2 ... word24"
```

Wallet names are normalized with the same rules as `wallet new` (lowercase, spaces to `-`, special chars removed).

`--secure` behavior is the same as in `wallet new`.
Saved config format is also the same as in `wallet new`, including `[wallets.<name>.expected].address-testnet`.
For global imports, Acton may create local `global.wallets.toml` symlink in project root for the current invocation.

### `acton wallet list`

Lists all available wallets (both local and global), their addresses, versions, and location (`[local]` / `[global]`).

```bash
acton wallet list
```

#### Options

| Option         | Type   | Description                               |
|----------------|--------|-------------------------------------------|
| `-b/--balance` | Flag   | Fetch and show wallet balances on testnet |
| `--api-key`    | String | Toncenter API key (optional)              |
| `--json`       | Flag   | Output result as JSON                     |

#### Example

```
$ acton wallet list --balance
Available wallets:
  deployer EQD... (v5r1) [local] — 2.4457 TON
  shared EQD... (v4r2) [global] — 0.5000 TON
```

If `--api-key` is omitted, Acton also checks the `TONCENTER_API_KEY` environment variable.

### `acton wallet export-mnemonic`

Exports wallet mnemonic phrase.

```bash
acton wallet export-mnemonic [NAME]
```

For security reasons:
- export is allowed only in interactive mode (`TTY`)
- command asks you to type wallet name to confirm export

Acton reads mnemonic from any configured source (`mnemonic`, `mnemonic-keyring`, `mnemonic-env`, `mnemonic-file`).

#### Example

```bash
acton wallet export-mnemonic deployer
```

### `acton wallet sign`

Signs wallet `external_body` BoC with the selected wallet private key using wallet-version-specific format.

```bash
acton wallet sign [NAME]
```

Input format is auto-detected:
- `hex` (preferred when a payload can be interpreted as both `hex` and `base64`)
- `base64`

Output is always signed `external_body` BoC in `hex`.

#### Options

| Option         | Type   | Description                                                    |
|----------------|--------|----------------------------------------------------------------|
| `--body <BOC>` | String | External body BoC to sign (auto-detected as `hex` or `base64`) |
| `--message <BOC>` | String | Alias for `--body`                                             |
| `--json`       | Flag   | Output result as JSON                                          |

If `--body` / `--message` is omitted, Acton will ask for the external body interactively.

#### Examples

```bash
# Sign external body provided as hex BoC
acton wallet sign deployer --body "B5EE9C72..."

# Sign external body provided as base64 BoC
acton wallet sign deployer --body "te6ccgEBAQEA..."
```

### `acton wallet airdrop`

Requests TON from a faucet for the selected wallet.

```bash
acton wallet airdrop [NAME]
```

With the default `--net testnet`, Acton requests a PoW challenge from faucet, solves it locally, and submits a claim for the selected wallet address.
With `--net localnet`, Acton sends a fixed `100 TON` airdrop through the local LiteNode admin faucet using `[litenode].port`.
Acton retries transient faucet errors (network/5xx) automatically with short backoff for the testnet backend.
If `[NAME]` is omitted, Acton auto-selects a wallet (single wallet) or asks you to choose (multiple wallets).

#### Options

| Option                 | Type   | Description                                                                  |
|------------------------|--------|------------------------------------------------------------------------------|
| `--net <NETWORK>`      | Enum   | Airdrop backend: `testnet` (default) or `localnet`                           |
| `--faucet-url <URL>`   | String | Faucet URL for `--net testnet` (default: `https://acton.monster/faucet/`)    |
| `--json`               | Flag   | Output result as JSON                                                        |

`--faucet-url` validation rules:
- only `http` / `https` schemes are allowed
- query parameters and URL fragments are not allowed
- cannot be used together with `--net localnet`

#### Examples

```bash
# Default testnet backend
acton wallet airdrop deployer --faucet-url https://your-faucet.example

# Localnet backend via LiteNode faucet
acton wallet airdrop deployer --net localnet
```

When `--json` is used:

#### Success JSON

| Field     | Type    | Description                                                |
|-----------|---------|------------------------------------------------------------|
| `success` | Boolean | Always `true` for successful airdrop                       |
| `message` | String  | Human-readable success message                             |
| `address` | String  | Wallet address used for claim                              |

For `--net testnet`, success JSON also includes:

| Field        | Type   | Description                                      |
|--------------|--------|--------------------------------------------------|
| `difficulty` | Number | PoW difficulty returned by faucet                |
| `nonce`      | Number | Solved PoW nonce                                 |
| `solve_ms`   | Number | Time spent solving PoW challenge in milliseconds |

#### Error JSON

| Field     | Type    | Description                                         |
|-----------|---------|-----------------------------------------------------|
| `success` | Boolean | Always `false` for failed airdrop                   |
| `error`   | String  | Human-readable error message describing the failure |

### `acton wallet remove`

Safely removes a wallet from configuration.

```bash
acton wallet remove [NAME]
```

If the wallet uses `mnemonic-keyring`, Acton also removes the mnemonic from your system keyring/keychain.
By default, Acton asks for confirmation and shows that this action cannot be undone.

#### Options

| Option        | Type | Description                                  |
|---------------|------|----------------------------------------------|
| `-y`, `--yes` | Flag | Skip confirmation prompt (for CI/scripts)    |
| `--json`      | Flag | Output result as JSON                        |

#### Examples

```bash
# Remove wallet by name
acton wallet remove deployer

# Interactive mode (if name is omitted)
acton wallet remove

# JSON output
acton wallet remove deployer --json -y
```

#### Notes

- If both local and global wallets have the same name, local `wallets.toml` takes precedence and is removed first.
- Removing a wallet from config does not remove external files referenced via `mnemonic-file` or environment variables.
- In non-interactive mode, `-y/--yes` is required to proceed.

## Getting Testnet Coins

Coins in testnet have no value, but you need them to pay for transaction fees (gas) when deploying contracts or sending messages.

To get testnet coins:

1.  Use the **[Testnet Giver Bot](https://t.me/testgiver_ton_bot)** on Telegram.
2.  Send your wallet address to the bot.
3.  The bot will send you some testnet TON.

For more methods (like web-based faucets), check the [official documentation](https://docs.ton.org/ecosystem/wallet-apps/get-coins#how-to-get-coins-on-testnet).

## Security

<Callout type="info">
    **Recommended: Secure Keyring**

    Acton supports storing mnemonics in your system's secure native store (Keyring/Keychain). This is the recommended way to handle wallet keys as they are not stored in plain text.
</Callout>

<Callout type="error">
    **SECURITY WARNING (Plain Text Storage)**

    If you choose not to use the secure keyring, the generated mnemonic is stored in **plain text** directly in the configuration file (`wallets.toml` or global config).

    - **NEVER** commit `wallets.toml` or `global.wallets.toml` to version control (git) if they contain plain text mnemonics.
    - Acton automatically adds these files to your `.gitignore` when initializing a project, but you should verify this.
    - These wallets are intended for **development and testing only**.
    - Do NOT use these wallets for mainnet funds or valuable assets.
</Callout>
